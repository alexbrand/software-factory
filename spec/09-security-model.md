# 09 — Security Model

**Status:** DRAFT

## Overview

The system runs untrusted AI-generated code inside sandboxes. The security model must assume that agent behavior is unpredictable and potentially malicious — not because the agents are adversarial, but because they execute arbitrary code generated by LLMs, which can be manipulated via prompt injection or simply make mistakes.

## Threat Model

### Actors

| Actor | Trust Level | Capabilities |
|-------|-------------|--------------|
| Platform operator | Trusted | Full cluster access, deploys system |
| Tenant admin | Semi-trusted | Manages pools, workflows in their namespace |
| Agent (in sandbox) | Untrusted | Executes arbitrary code, makes network calls |
| External client | Untrusted | Submits workflows via API |

### Threats

| Threat | Risk | Mitigation |
|--------|------|------------|
| Sandbox escape | Critical | Container isolation, seccomp, AppArmor, optional gVisor |
| Cross-tenant data access | Critical | Namespace isolation, network policies, separate PVCs |
| Secret exfiltration | High | Credential proxy, never expose raw keys to agent context |
| Network-based attacks | High | Egress-only network policies, allowlisted destinations |
| Resource exhaustion (DoS) | Medium | Resource quotas, LimitRanges, task timeouts |
| Prompt injection via artifacts | Medium | Artifact validation, content scanning |
| Supply chain (dependency) | Medium | Image scanning, signed images, read-only base layers |
| MCP tool abuse | Medium | Tool-level RBAC via ToolHive vMCP, per-tenant tool curation |

## Isolation Layers

### Layer 1: Kubernetes Namespace Isolation

Each tenant gets a dedicated namespace with:

```yaml
# ResourceQuota per tenant
apiVersion: v1
kind: ResourceQuota
metadata:
  name: tenant-quota
  namespace: team-alpha
spec:
  hard:
    pods: "50"
    requests.cpu: "100"
    requests.memory: "200Gi"
    persistentvolumeclaims: "100"
    requests.storage: "500Gi"
```

### Layer 2: Pod Security

Sandbox pods run with restricted security contexts:

```yaml
securityContext:
  runAsNonRoot: true
  runAsUser: 1000
  readOnlyRootFilesystem: false  # Agents need to write
  allowPrivilegeEscalation: false
  capabilities:
    drop: [ALL]
    add: [NET_BIND_SERVICE]  # Only if needed
  seccompProfile:
    type: RuntimeDefault
```

For higher security requirements, deploy with:
- **gVisor (runsc)**: Application kernel that intercepts syscalls. Prevents most container escape vectors.
- **Kata Containers**: Lightweight VMs for hardware-level isolation.

These are configured via Kubernetes RuntimeClass:

```yaml
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: gvisor
handler: runsc

---
# Referenced in Pool spec:
spec:
  sandboxTemplate:
    runtimeClassName: gvisor
```

### Layer 3: Network Isolation

Each sandbox gets a NetworkPolicy restricting network access:

1. **Deny all by default**: No ingress or egress without explicit rules.
2. **Allow DNS**: Required for name resolution.
3. **Allow specific egress**: Only allowlisted destinations (LLM APIs, git hosts, package registries).
4. **Allow control plane ingress**: The bridge sidecar HTTP port is accessible only from the Session Controller.

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: sandbox-default-deny
spec:
  podSelector:
    matchLabels:
      factory.example.com/component: sandbox
  policyTypes: [Ingress, Egress]
  ingress:
    - from:
        - podSelector:
            matchLabels:
              factory.example.com/component: session-controller
      ports:
        - port: 8080
  egress:
    - to:
        - namespaceSelector: {}
      ports:
        - port: 53
          protocol: UDP
    # Additional egress rules from Pool spec
```

### Layer 4: Credential Isolation

Agents must never have direct access to raw credentials. The credential proxy sidecar handles this:

```
Agent → HTTP request to api.anthropic.com → Credential Proxy → Injects Authorization header → External API
```

**How it works:**
1. Agent's HTTP traffic is routed through the sidecar proxy (via environment variables: `HTTP_PROXY`/`HTTPS_PROXY`).
2. The proxy matches outbound requests against a credential mapping:
   ```yaml
   credentialMappings:
     - host: "api.anthropic.com"
       header: "x-api-key"
       secretRef:
         name: anthropic-credentials
         key: api-key
     - host: "github.com"
       header: "Authorization"
       value: "token ${GITHUB_TOKEN}"
   ```
3. Credentials are injected into the request at the proxy layer.
4. The agent environment contains no API keys — only the proxy URL.

This is inspired by Cloudflare Dynamic Workers' `globalOutbound` pattern.

### Layer 5: MCP Tool-Level Access Control

When agents use MCP tools (databases, APIs, external services), access is controlled at the Pool level:

1. **Per-tenant tool curation**: Each Pool defines the specific MCP server endpoints its sandboxes can access via `mcpServers`. Agents only see tools from those endpoints — not every MCP server in the cluster.
2. **Secret isolation**: MCP server credentials (database passwords, API tokens) are managed by the MCP server deployment, not by the agent sandbox. The agent calls tools via the MCP protocol — it never sees the credentials the MCP server uses to connect to backends.
3. **Network isolation**: Sandbox NetworkPolicies restrict which MCP endpoints are reachable. An agent can only call tools at the endpoints listed in its Pool.

For production deployments with many MCP servers, [ToolHive](https://github.com/stacklok/toolhive) provides additional capabilities: tool aggregation via VirtualMCPServer (vMCP), tool-level RBAC, conflict resolution across servers, audit logging of tool invocations, and token optimization to reduce the attack surface from prompt injection.

## RBAC

### System Roles

| Role | Scope | Permissions |
|------|-------|-------------|
| `factory-admin` | Cluster | Full access to all resources |
| `tenant-admin` | Namespace | Manage pools, agent configs, view all workflows |
| `developer` | Namespace | Submit workflows, view own tasks and sessions |
| `viewer` | Namespace | Read-only access to workflows and sessions |

### Kubernetes RBAC Integration

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: developer
  namespace: team-alpha
rules:
  - apiGroups: ["factory.example.com"]
    resources: ["workflows", "tasks"]
    verbs: ["create", "get", "list", "watch", "delete"]
  - apiGroups: ["factory.example.com"]
    resources: ["sessions"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["factory.example.com"]
    resources: ["pools", "sandboxes"]
    verbs: ["get", "list"]
```

### API Authentication

The API server supports:
1. **Bearer tokens**: Kubernetes ServiceAccount tokens or external OIDC tokens.
2. **mTLS**: For service-to-service communication.
3. **Impersonation**: The API server impersonates the authenticated user when creating K8s resources, inheriting their RBAC permissions.

## Secrets Management

### Hierarchy

```
Cluster Secrets (managed by platform operator)
├── LLM API keys (shared across tenants or per-tenant)
├── Container registry credentials
└── TLS certificates

Tenant Secrets (managed by tenant admin)
├── Git tokens (per-repository)
├── External API keys (per-service)
└── Webhook signing keys

Task Secrets (injected per-task)
├── Scoped credentials from tenant secrets
└── Ephemeral tokens (e.g., short-lived git tokens)
```

### Secret Injection Flow

1. `AgentConfig` declares required credentials.
2. `Pool` spec references secrets by `secretRef`.
3. Sandbox Controller mounts secrets as projected volumes.
4. Credential proxy reads secrets from the volume.
5. Agents see only the proxy URL, never the secret values.

### External Secrets

For production deployments, integrate with external secret stores via the [External Secrets Operator](https://external-secrets.io/) (a CNCF project):

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: anthropic-credentials
  namespace: team-alpha
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend
    kind: ClusterSecretStore
  target:
    name: anthropic-credentials
  data:
    - secretKey: api-key
      remoteRef:
        key: secret/data/anthropic
        property: api-key
```

## Audit Trail

All security-relevant events are logged:

- Authentication attempts (success/failure)
- RBAC authorization decisions
- Secret access (which pod accessed which secret)
- Network policy violations (via CNI logging)
- Sandbox lifecycle events (creation, termination, privilege changes)

Audit logs are forwarded to the event bus and can be exported to a SIEM system.

## Image Security

- **Base images**: Maintained by the platform team, rebuilt weekly with security patches.
- **Image scanning**: All images scanned with Trivy (or equivalent) in CI.
- **Signed images**: Use cosign/sigstore for image signatures. Admission controller (Kyverno or OPA Gatekeeper) enforces signature verification.
- **Immutable tags**: Sandbox images use digest-based references, not mutable tags.
